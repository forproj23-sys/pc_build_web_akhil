PC BUILDING WEBSITE – DATABASE SCHEMA DOCUMENTATION
================================================================================

This document contains the complete database schema structure for the PC Building
Website project, including all collections (tables) and their fields.

================================================================================
1. USER COLLECTION
================================================================================

Collection Name: users
Description: Stores user account information including authentication details
             and role-based access control.

┌──────────────────────┬──────────────┬──────────────────────────────────────┬──────────┬─────────────────────────────────────────────┐
│ Field Name           │ Data Type    │ Constraints                          │ Required │ Description                                 │
├──────────────────────┼──────────────┼──────────────────────────────────────┼──────────┼─────────────────────────────────────────────┤
│ _id                  │ ObjectId     │ Primary Key, Auto-generated          │ Yes      │ Unique identifier for each user            │
│ name                 │ String       │ trim: true                           │ Yes      │ User's full name                           │
│ email                │ String       │ unique: true, lowercase: true,        │ Yes      │ User's email address (unique)               │
│                      │              │ trim: true                           │          │                                             │
│ password             │ String       │ Hashed with bcrypt                   │ Yes      │ Encrypted password for authentication      │
│ role                 │ String       │ enum: ['user', 'admin',              │ No       │ User role in the system                    │
│                      │              │ 'assembler', 'supplier'],            │          │ Default: 'user'                            │
│                      │              │ default: 'user'                      │          │                                             │
│ approved             │ Boolean      │ default: false                       │ No       │ Admin approval status for account access    │
│ createdAt            │ Date         │ Auto-generated                       │ Yes      │ Timestamp when user was created             │
│ updatedAt            │ Date         │ Auto-generated                       │ Yes      │ Timestamp when user was last updated        │
└──────────────────────┴──────────────┴──────────────────────────────────────┴──────────┴─────────────────────────────────────────────┘

3. BUILD COLLECTION
================================================================================

Collection Name: builds
Description: Stores PC build configurations created by users, including
             component selections, assembly status, payment information,
             and refund requests.

================================================================================
RELATIONSHIPS SUMMARY
================================================================================

1. User → Component (One-to-Many)
   - supplierID in Component references User._id
   - A supplier user can have multiple components

2. User → Build (One-to-Many)
   - userID in Build references User._id
   - A user can create multiple builds

3. User → Build (One-to-Many) - Assembler Assignment
   - assemblerID in Build references User._id
   - An assembler can be assigned to multiple builds

4. Component → Build (Many-to-Many)
   - components array in Build contains componentID references
   - A build contains multiple components
   - A component can be used in multiple builds

5. Build → Transaction (One-to-Many)
   - buildId in Transaction references Build._id
   - A build can have multiple transactions (payments, refunds)

6. User → Build → Supplier Payouts (Many-to-Many)
   - supplierPayouts array in Build contains supplierID references
   - A build can have payouts to multiple suppliers
   - A supplier can receive payouts from multiple builds

================================================================================
INDEXES
================================================================================

Automatically created indexes:
- User.email: Unique index for fast email lookups
- Category.name: Unique index for category name lookups
- Build.userID: Index for querying builds by user
- Build.assemblerID: Index for querying builds by assembler
- Component.supplierID: Index for querying components by supplier
- Transaction.buildId: Index for querying transactions by build

================================================================================
TIMESTAMPS
================================================================================

All collections include automatic timestamps:
- createdAt: Automatically set when document is created
- updatedAt: Automatically updated when document is modified

These fields are managed by Mongoose and do not need to be manually set.
